# 痛点根因拆解方法论

## 为什么要做根因拆解？

差评痛点如果只停留在用户感受层（"漏水""蒸汽弱"），供应商沟通会变成模糊需求。**拆解到原理层**后变成精确的工程要求，工厂一听就能执行。

## 拆解流程（5 步）

### Step 1：归集差评原文
以**物理现象**（不是用户措辞）为分组依据。例如"漏水"可能是：密封圈老化、壳体开裂、冷水喷射、配件接口松动——4种完全不同的问题。

### Step 2：画出产品工作原理
标注能量/物质流动路径、关键连接界面、运动/受力部件。可 WebSearch "[product type] teardown" 了解内部结构。

### Step 3：物理归因（5 Whys）
```
用户说："It spits water"（喷水）
  Why? → 液态水从喷嘴喷出
  Why? → 水在加热腔内未完全汽化
  Why? → 加热腔容积小 + 无气液分离机构
  Why? → 成本设计取舍，基础款省掉了分离结构
  → 根因：缺少气液分离机构
```

### Step 4：区分「品类限制」vs「设计缺陷」

| 类型 | 定义 | MVP 能否解决 | 应对策略 |
|------|------|-------------|---------|
| **设计缺陷** | 竞品偷工减料导致 | ✅ 可以 | 定向改良，纳入 MVP 规格 |
| **品类限制** | 受物理定律决定的上限 | ❌ 不能 | Listing 中管理期望 |
| **使用问题** | 用户操作不当 | ⚠️ 部分 | 改善说明书/引导设计 |

### Step 5：输出工程方案 + 供应商话术
列出 2-4 种方案，评估难度和增量成本，推荐 MVP 最优组合。

## 输出模板

```
## 痛点 X — [用户感知名称]

### 用户原文
> "xxx" / "yyy"

### Review 双向提及率
- 差评提及率 (complaint_rate)：XX% — 从 Phase 2 主报告提取
- 好评提及率 (positive_mention_rate)：XX% — 从 Phase 2 CSV 统计

> 数据来源：complaint_rate 和 positive_mention_rate 优先从 Phase 2 上游提取，不重复搜索。
> 这是 KANO 分类的关键输入——高差评+低好评=基本型(M)；双高=期望型(O)；低差评+高好评=兴奋型(A)。

### 5 Whys 归因
- 现象 → Why1 → Why2 → **根因**
- 类型：设计缺陷 / 品类限制 / 使用问题

### 工程解决方案
| 方案 | 原理 | 难度 | 增量成本 |

### 供应商沟通话术
- ❌ 不要说："我要产品不XXX"
- ✅ 正确说法："[具体工程改良要求]"
```

## 常见品类参考

| 品类 | 典型痛点 | 常见根因方向 |
|------|---------|-------------|
| 电器类（蒸汽/加热） | 漏水、过热、功率不足 | 密封材质、热管理、电路保护 |
| 塑料外壳类 | 开裂、变色、异味 | 材质等级、壁厚、注塑工艺 |
| 可拆卸配件类 | 松动、难装、断裂 | 连接方式、公差控制 |
| 电机类 | 噪音大、振动、寿命短 | 电机类型、轴承选型、减震 |

## 供应商沟通原则

1. **用工程语言**：不说"蒸汽弱"，说"加热腔增加翅片面积提升热交换效率"
2. **给量化指标**：不说"密封要好"，说"EPDM 硅胶，硬度 50±5 Shore A，耐温 ≥200°C"
3. **区分 P0/P1/P2**：让工厂知道哪些是底线
4. **给竞品参照物**："参考 [竞品ASIN]，但在 [部位] 需改成 [方案]"
