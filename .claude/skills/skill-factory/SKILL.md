# Skill Factory — MECE 增强型技能创建器

在 Anthropic 官方 skill-creator 的完整能力基础上，增加 **MECE 分析法**（Mutually Exclusive, Collectively Exhaustive），确保每一个 Skill 都经过系统化的要素穷举和关系梳理。

## 核心理念

> 一个好的 Skill 不是写出来的，是**分析**出来的。

在动手写 SKILL.md 之前，先用 MECE 把"这个 Skill 需要覆盖什么"彻底想清楚。这避免了最常见的问题：Skill 写好了才发现漏了关键场景、忘了边界情况、或者职责定义含糊。

---

## 工作流程总览

```
1. 捕获意图 → 2. MECE 分析 → 3. 要素确认 → 4. 编写 SKILL.md → 5. 测试评估 → 6. 迭代改进 → 7. 描述优化 → 8. 打包发布
```

比标准 skill-creator 多了一个关键步骤：**MECE 分析**。

---

## Phase 0: 捕获意图（同 skill-creator）

开始时理解用户的意图：

1. 这个 Skill 要让 Claude 做什么？
2. 什么时候应该触发？（用户会说什么话？）
3. 期望的输出格式是什么？
4. 需要设置测试用例吗？

如果对话中已经有工作流（用户说"把这个变成 skill"），从对话历史中提取答案。

---

## Phase 1: MECE 分析 ⭐ 核心增强

这是本 Skill 区别于标准 skill-creator 的核心。在写任何代码之前，先对"这个 Skill 要解决的问题空间"做 MECE 分解。

### Step 1: 定义问题空间

明确这个 Skill 的**一级边界**：
- 输入类型有哪些？
- 输出类型有哪些？
- 用户场景有哪些？
- 依赖什么工具/数据？

### Step 2: MECE 分解 — 穷尽所有要素

使用以下框架之一（根据 Skill 类型选择最合适的）：

#### 框架 A：场景分解（适合流程型 Skill）
```
问题空间
├── 场景1: [描述]
│   ├── 子场景1.1
│   └── 子场景1.2
├── 场景2: [描述]
│   ├── 子场景2.1
│   └── 子场景2.2
├── 场景3: [描述]
└── 场景4: [描述]
✅ 检查：这些场景是否覆盖了所有可能？是否有重叠？
```

#### 框架 B：维度分解（适合分析型 Skill）
```
维度1: [如：数据来源]
  ├── 选项A | 选项B | 选项C
维度2: [如：分析深度]
  ├── 快速扫描 | 标准分析 | 深度报告
维度3: [如：输出格式]
  ├── 文本 | 表格 | 卡片 | 文档
✅ 检查：各维度是否独立？组合是否穷尽？
```

#### 框架 C：价值链分解（适合端到端工作流 Skill）
```
输入 → 采集 → 处理 → 分析 → 决策 → 输出 → 反馈
每个环节：什么能出错？什么可以自动化？什么需要人工判断？
```

#### 框架 D：用户旅程分解（适合交互型 Skill）
```
新手用户路径:  触发 → 引导 → 执行 → 结果
老手用户路径:  触发 → 直接执行 → 结果
异常路径:     触发 → 错误 → 恢复/提示
```

### Step 3: 绘制要素关系图

识别要素之间的**依赖、冲突和协同**关系：

```
[要素A] ──依赖──→ [要素B]
[要素C] ←─冲突──→ [要素D]
[要素E] ←─协同──→ [要素F]
```

问自己：
- 哪些要素必须按顺序执行？（依赖）
- 哪些要素不能同时存在？（互斥）
- 哪些要素组合起来效果更好？（协同）

### Step 4: 生成选择矩阵

将分解结果转化为**用户可选择的选项**，让用户参与决策：

```
┌─────────────────────────────────────────────┐
│  Skill 设计选择矩阵                          │
├──────────┬──────────┬──────────┬─────────────┤
│ 维度      │ 选项 A   │ 选项 B   │ 推荐        │
├──────────┼──────────┼──────────┼─────────────┤
│ 触发方式  │ 关键词    │ 上下文    │ 两者结合 ✅  │
│ 输出深度  │ 简报      │ 完整报告  │ 视场景而定   │
│ 数据源    │ MCP 工具  │ 用户输入  │ MCP 优先 ✅  │
│ 交互模式  │ 一次性    │ 多轮对话  │ 取决于复杂度  │
└──────────┴──────────┴──────────┴─────────────┘
```

### Step 5: MECE 检查清单

在进入编写阶段前，过一遍这个检查清单：

- [ ] **互斥性（ME）**：每个分类之间没有重叠吗？
- [ ] **穷尽性（CE）**：所有可能的情况都覆盖了吗？
- [ ] **边界情况**：空输入、异常输入、极端情况考虑了吗？
- [ ] **错误处理**：每种失败模式都有应对方案吗？
- [ ] **依赖关系**：所有需要的工具/MCP/API 都确认可用了吗？
- [ ] **用户确认**：选择矩阵已经给用户看过了吗？

> 向用户展示 MECE 分解结果和选择矩阵，获得确认后再进入编写阶段。

## Phase 2: 编写 SKILL.md

### Skill 目录结构

```
skill-name/
├── SKILL.md (必需)
│   ├── YAML frontmatter (name, description 必需)
│   └── Markdown 指令
└── 附属资源 (可选)
    ├── scripts/    - 可执行脚本
    ├── references/ - 需要时加载的文档
    └── assets/     - 输出中使用的文件（模板、图标等）
```

### 三级加载系统

1. **元数据**（name + description）— 始终在上下文中（~100 词）
2. **SKILL.md 正文** — 技能触发时加载（理想 <500 行）
3. **附属资源** — 按需加载（无限制，脚本可直接执行无需加载）

### 编写原则

- **SKILL.md 控制在 500 行以内**；超过时增加层级结构和引用
- **参考文件从 SKILL.md 中明确引用**，说明何时需要读取
- **大型参考文件（>300 行）包含目录**
- **用祈使句写指令**
- **解释 why，而不是堆 MUST** — 让模型理解为什么重要比强制命令更有效
- **描述要"主动"一些** — 倾向于宁可多触发也不要漏触发

### 输出格式定义

可以这样定义：
```markdown
## 报告结构
始终使用此模板：
# [标题]
## 执行摘要
## 核心发现
## 建议
```

### 示例模式

包含示例很有用：
```markdown
## 提交格式
**示例 1:**
输入: 添加了 JWT 用户认证
输出: feat(auth): 实现基于 JWT 的认证
```

## Phase 3: 测试与评估

### 创建测试用例

编写 2-3 个现实的测试提示。与用户确认后运行。

保存到 `evals/evals.json`：
```json
{
  "skill_name": "example-skill",
  "evals": [
    {
      "id": 1,
      "prompt": "用户的任务提示",
      "expected_output": "预期结果描述",
      "files": []
    }
  ]
}
```

### 运行测试

对每个测试用例，同时启动两个子代理：
- **使用技能的运行**：读取 SKILL.md 后执行任务
- **基线运行**：不使用技能直接执行（新建技能时），或使用旧版本（改进技能时）

结果保存到 `<skill-name>-workspace/iteration-<N>/eval-<ID>/` 目录。

### 评分与基准测试

1. **评分** — 使用 `agents/grader.md` 代理评估每个断言
2. **汇总** — 运行 `scripts/aggregate_benchmark.py` 生成基准统计
3. **分析** — 使用 `agents/analyzer.md` 发现聚合指标隐藏的模式
4. **可视化** — 使用 `eval-viewer/generate_review.py` 展示结果

```bash
# 汇总基准数据
python -m scripts.aggregate_benchmark <workspace>/iteration-N --skill-name <name>

# 启动查看器
nohup python <skill-factory-path>/eval-viewer/generate_review.py \
  <workspace>/iteration-N \
  --skill-name "my-skill" \
  --benchmark <workspace>/iteration-N/benchmark.json \
  > /dev/null 2>&1 &
```

对于无浏览器环境，使用 `--static <output_path>` 生成独立 HTML。

### 用户反馈

用户完成评审后，读取 `feedback.json`：
```json
{
  "reviews": [
    {"run_id": "eval-0-with_skill", "feedback": "图表缺少坐标轴标签", "timestamp": "..."}
  ],
  "status": "complete"
}
```

空反馈表示用户认为没问题。聚焦有具体意见的测试用例。

## Phase 4: 迭代改进

### 改进原则

1. **从反馈中泛化** — 不要过拟合到测试用例，让改进惠及所有未来用户
2. **保持精简** — 删掉没用的部分。读转录记录，看模型是否在做无效工作
3. **解释 why** — 让模型理解为什么这样做重要，而不是堆规则
4. **发现重复模式** — 如果多个测试都写了类似的辅助脚本，把它打包成 `scripts/`

### 迭代循环

1. 应用改进
2. 重新运行所有测试（新的 `iteration-<N+1>/` 目录）
3. 用 `--previous-workspace` 启动查看器展示对比
4. 等待用户反馈
5. 继续改进，直到用户满意

---

## Phase 5: 描述优化（可选）

优化 description 字段以提高触发准确率。

### 生成 20 个评估查询
- 8-10 个应该触发的（各种表达方式）
- 8-10 个不应该触发的（相似但不同的场景）

查询要**真实且具体**，包含文件路径、个人背景、具体细节。不要写抽象请求。

### 运行优化循环
```bash
python -m scripts.run_loop \
  --eval-set <path-to-trigger-eval.json> \
  --skill-path <path-to-skill> \
  --model <model-id> \
  --max-iterations 5 \
  --verbose
```

```bash
python -m scripts.package_skill <path/to/skill-folder>
```

## MECE 分析的实战示例

### 示例：为"亚马逊竞品监控"设计 Skill

**Step 1 — 问题空间：**
监控亚马逊竞品的价格/排名/评论变化，有异常时通知用户

**Step 2 — MECE 分解：**
```
竞品监控 Skill
├── 数据维度（监控什么）
│   ├── 价格变动
│   ├── BSR 排名变化
│   ├── 评论数增速
│   ├── Listing 变更（标题/图片/五点）
│   └── 上新/下架
├── 触发维度（什么时候监控）
│   ├── 手动触发（用户说"查一下"）
│   ├── 定时触发（每天/每周）
│   └── 事件触发（价格变动超阈值）
├── 输出维度（怎么通知）
│   ├── 飞书卡片
│   ├── 飞书文档
│   ├── 终端输出
│   └── 邮件（如果有）
└── 竞品维度（监控谁）
    ├── 指定 ASIN 列表
    ├── 类目 Top N
    └── 同关键词竞争者
```

**Step 3 — 关系图：**
```
[定时触发] ──依赖──→ [数据源 API]
[飞书通知] ──依赖──→ [飞书 Webhook 配置]
[类目 Top N] ←─协同──→ [BSR 排名变化] （都需要类目数据）
[手动触发] ←─互斥──→ [定时触发] （同一次只能一种）
```

**Step 4 — 选择矩阵呈现给用户**

这就是用 MECE 分析 Skill 的完整过程。通过这种方式，你不会漏掉任何关键场景。

## 与用户沟通

注意用户的技术水平。如果用户不是程序员：
- "evaluation"和"benchmark"可以用，但解释一下
- "JSON"和"assertion"需要先确认用户理解
- 简要定义术语，不确定时宁可多解释

## 参考文件

agents/ 目录包含专门的子代理指令：
- `agents/grader.md` — 如何评估断言
- `agents/comparator.md` — 如何做盲 A/B 对比
- `agents/analyzer.md` — 如何分析为什么某个版本更好

references/ 目录有附加文档：
- `references/schemas.md` — evals.json、grading.json 等 JSON 结构定义

## 核心循环强调

1. 搞清楚 Skill 要做什么
2. **MECE 分析穷尽要素**（本工厂的核心增强）
3. 编写或编辑 Skill
4. 在测试提示上运行 claude-with-skill
5. 与用户一起评估输出（定性 + 定量）
6. 基于反馈改进
7. 重复直到满意
8. 打包最终 Skill

Good luck! 🏭
