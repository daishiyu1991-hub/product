# Impact x Effort 优先级评估框架

## 概述

本框架用于在每个迭代周期中对候选优化项进行系统化评估，确保团队始终将精力投入到投入产出比最高的改进上。所有优化项在进入执行阶段前，必须经过 Impact x Effort 评分。

---

## 一、Impact（影响力）评分标准

根据预期对核心业务指标（转化率、销量、利润率、排名）的影响程度打分：

| 分数 | 等级 | 定义 | 示例 |
|------|------|------|------|
| **1** | 微小 | 几乎无法察觉的改善，对关键指标影响 < 1% | 修改 bullet point 中的一个标点符号 |
| **2** | 轻微 | 可能带来小幅提升，对关键指标影响 1%-3% | 优化后台搜索词中的少量关键词 |
| **3** | 中等 | 有明确可衡量的提升，对关键指标影响 3%-10% | 调整定价策略、增加一个颜色变体 |
| **4** | 显著 | 带来明显业务改善，对关键指标影响 10%-25% | 标题 A/B 测试优化、主要关键词排名提升 |
| **5** | 重大 | 带来根本性改变，对关键指标影响 > 25% | 主图全面重新设计、产品功能升级 |

### Impact 评估维度

评估 Impact 时，综合考虑以下维度：

1. **转化率影响** — 该优化是否直接影响买家购买决策？
2. **流量影响** — 该优化是否能带来更多曝光和点击？
3. **利润影响** — 该优化是否影响单位利润或整体利润率？
4. **持续性** — 该优化的效果是短期的还是长期的？
5. **覆盖范围** — 该优化影响一个 ASIN 还是多个 ASIN？

---

## 二、Effort（工作量）评分标准

根据完成该优化项所需的时间、资源和复杂度打分：

| 分数 | 等级 | 时间估算 | 所需资源 | 示例 |
|------|------|----------|----------|------|
| **1** | 极小 | ≤ 1 小时 | 仅需卖家自己操作 | 修改标题关键词、添加否定关键词 |
| **2** | 较小 | 1-4 小时 | 可能需要简单工具辅助 | 撰写新的五点描述、调整广告竞价 |
| **3** | 中等 | 1-3 天 | 需要协调设计师或其他成员 | 制作新副图、创建 A+ 内容 |
| **4** | 较大 | 1-2 周 | 需要多人协作或外部供应商 | 重新拍摄产品照片、开发新包装 |
| **5** | 巨大 | 1 个月以上 | 需要重大投资或供应链变更 | 产品开模改款、开发全新变体 |

### Effort 评估维度

评估 Effort 时，综合考虑以下维度：

1. **时间成本** — 从启动到完成需要多长时间？
2. **资金成本** — 需要额外投入多少费用？
3. **人力需求** — 需要几个人参与？是否需要外部资源？
4. **技术难度** — 执行过程是否有技术障碍？
5. **风险程度** — 失败的概率和失败后的恢复成本？

---

## 三、优先级计算公式

### 基础公式

```
优先级分数 = Impact 分数 / Effort 分数
```

### 加权调整公式（推荐）

当需要更精细的排序时，使用加权公式：

```
优先级分数 = (Impact 分数 × 置信系数) / Effort 分数
```

**置信系数**定义：
- **1.0** — 有历史数据或测试数据支持
- **0.8** — 基于行业经验和竞品分析推断
- **0.6** — 基于直觉判断，缺乏数据支持

### 优先级分数解读

| 优先级分数 | 行动建议 |
|------------|----------|
| ≥ 3.0 | 立即执行，列入当前迭代 |
| 2.0 - 2.9 | 高优先级，列入下一迭代 |
| 1.0 - 1.9 | 中等优先级，排期执行 |
| < 1.0 | 低优先级，暂时搁置或放弃 |

## 四、四象限分类法

根据 Impact 和 Effort 的高低组合，将所有优化项划分为四个象限：

```
                    ┌─────────────────────────────────────────┐
                    │                                         │
        Impact 高   │   🏗️ High Value     │   🎯 Quick Win    │
        (4-5分)     │   (高影响·高投入)    │   (高影响·低投入)  │
                    │   → 仔细规划后执行   │   → 立即执行       │
                    │                                         │
                    ├─────────────────────────────────────────┤
                    │                                         │
        Impact 低   │   🚫 Avoid          │   📝 Fill In      │
        (1-3分)     │   (低影响·高投入)    │   (低影响·低投入)  │
                    │   → 不做             │   → 有空时做       │
                    │                                         │
                    └─────────────────────────────────────────┘
                          Effort 高              Effort 低
                          (4-5分)               (1-2分)
```

### 各象限详细策略

#### 🎯 Quick Win（速赢项）— 高影响 · 低投入

- **特征**：Impact ≥ 4, Effort ≤ 2
- **策略**：立即执行，不需要过多讨论
- **执行方式**：直接放入当前迭代周期，优先完成
- **典型案例**：标题关键词优化、价格微调、否定关键词清理
- **注意事项**：速赢项是团队士气的来源，确保每个迭代都有速赢项

#### 🏗️ High Value（高价值项）— 高影响 · 高投入

- **特征**：Impact ≥ 4, Effort ≥ 4
- **策略**：仔细规划后执行，拆分为多个小步骤
- **执行方式**：制定详细计划，分配专人负责，跨多个迭代周期完成
- **典型案例**：主图全面改版、产品功能升级、新变体开发
- **注意事项**：必须设定里程碑和检查点，避免资源黑洞

#### 📝 Fill In（填充项）— 低影响 · 低投入

- **特征**：Impact ≤ 3, Effort ≤ 2
- **策略**：有空闲时间时做，不占用主要精力
- **执行方式**：放入待办列表，在主要任务完成后见缝插针地执行
- **典型案例**：后台搜索词微调、次要副图更新、小幅文案修改
- **注意事项**：不要让填充项积累过多，定期清理

#### 🚫 Avoid（避免项）— 低影响 · 高投入

- **特征**：Impact ≤ 3, Effort ≥ 4
- **策略**：不做，除非有明确的战略考量
- **执行方式**：直接从待办列表移除，释放认知负担
- **典型案例**：低效益的产品改款、盲目跟风的功能添加
- **注意事项**：如果多个避免项可以合并执行降低总 Effort，可重新评估

## 六、迭代周期后的重新评估

### 每个迭代周期结束时的评估流程

每个迭代周期（通常为 1-2 周）结束后，必须对未完成项和新增项进行重新评估：

#### 步骤 1：回顾已完成项的实际效果

```
实际 Impact vs 预期 Impact 对比：
- 标题 A/B 测试 → 预期 Impact: 4, 实际: ___
- 价格调整     → 预期 Impact: 3, 实际: ___
```

对比实际效果与预期评分，校准团队的评估准确度。

#### 步骤 2：更新未完成项的评分

根据新获取的数据和市场变化，调整未完成项的 Impact 和 Effort 分数：

- 市场环境变化是否改变了某项优化的预期影响？
- 是否发现了更高效的执行方法，降低了 Effort？
- 是否有新的竞品动作改变了优先级？

#### 步骤 3：纳入新发现的优化机会

从以下来源收集新的优化项：

- 上一周期的数据分析中发现的新问题
- 竞品监控中发现的新机会
- 客户反馈/差评中暴露的痛点
- 行业趋势和季节性因素

#### 步骤 4：重新排序并制定下一周期计划

1. 将所有候选项按优先级分数降序排列
2. 确保每个迭代包含至少 1 个 Quick Win
3. 每个迭代最多安排 1 个 High Value 项（避免过载）
4. Fill In 项根据剩余产能灵活安排
5. 定期清理 Avoid 象限，移除不再考虑的项目

### 评估校准机制

每完成 3-5 个迭代周期后，进行一次系统性校准：

1. **准确度回顾**：统计预期 vs 实际的偏差率
2. **偏差分析**：团队是否系统性高估或低估某类优化的 Impact？
3. **Effort 校准**：实际执行时间是否与估算一致？
4. **调整评分标准**：根据历史数据更新评分基准

### 动态调整原则

- **数据优先**：当有实际数据时，用数据替代直觉判断
- **小批量验证**：对不确定的高 Impact 评分项，先做小规模测试
- **沉没成本无视**：已投入的 Effort 不应影响是否继续的决策
- **环境敏感**：大促前、旺季前、竞品变动时要及时重新评估
