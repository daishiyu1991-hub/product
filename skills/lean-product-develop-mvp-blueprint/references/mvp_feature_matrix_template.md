# MVP 功能矩阵模板与填写指南

## 概述

功能矩阵是 MVP 蓝图的核心工具，用于系统性地将产品属性维度分为 Must-have / Nice-to-have / Cut 三级，确保 MVP 产品"刚好够用来验证市场假设"，而非追求完美。

---

## 1. 三级分类标准

### 1.1 分类定义与量化阈值

| 分类 | 定义 | 量化阈值 | 决策逻辑 |
|------|------|----------|----------|
| **Must-have** | 没有就卖不动的功能 | ①差评提及率 ≥15%；**或** ②交叉分析中属于"高需求 × 薄供给"象限；**或** ③品类强制合规要求（认证/安规） | 不做 = 没有差异化 = 无法成交 |
| **Nice-to-have** | 锦上添花的功能 | ①差评提及率 5%~14%；**或** ②交叉分析中属于"有需求 × 有供给"象限；**或** ③搜索量 >5000/月但非核心购买维度 | 成本允许就做，超预算就 Cut |
| **Cut** | MVP 阶段明确不做 | ①差评提及率 <5%；**或** ②增加单位成本 > $1 但不影响核心购买决策；**或** ③需要额外认证/开模导致上市延迟 >30天 | 留给 v2 迭代 |

### 1.2 决策流程图

```
属性维度
  │
  ├─ 差评提及率 ≥15%? ──── 是 ──→ Must-have
  │     │
  │     否
  │     │
  ├─ 交叉分析"高需求×薄供给"? ── 是 ──→ Must-have
  │     │
  │     否
  │     │
  ├─ 品类合规强制要求? ──── 是 ──→ Must-have
  │     │
  │     否
  │     │
  ├─ 差评提及率 5~14%? ──── 是 ──→ Nice-to-have（检查成本）
  │     │                              │
  │     │                         增加成本 > 预算上限?
  │     │                         ├── 是 ──→ Cut
  │     │                         └── 否 ──→ 保留 Nice-to-have
  │     否
  │     │
  └─ 差评提及率 <5% ──────────────→ Cut
```

## 4. MVP 范围控制规则

### 4.1 硬性约束

| 规则 | 标准 | 说明 |
|------|------|------|
| ⛔ Must-have 数量上限 | **5~7 个** | 超过 7 个说明范围失控，需重新评估优先级 |
| ⛔ Must-have 数量下限 | **3 个** | 少于 3 个说明差异化不足，需补充竞争维度 |
| ⛔ 单项增量成本上限 | **≤ 目标售价的 8%** | 单个 Nice-to-have 增量成本超限必须 Cut |
| ⛔ Must-have 总增量成本 | **≤ 目标毛利的 40%** | 超限需要提高售价或砍减功能 |
| ⛔ 首批 SKU 数量 | **≤ 3 个**（颜色/尺寸变体） | MVP 阶段控制库存风险 |

### 4.2 范围蔓延预警信号

| 预警信号 | 触发条件 | 应对措施 |
|----------|----------|----------|
| 🟡 Must-have 超过 7 个 | 团队想把所有功能都做成 Must-have | 强制排序，仅保留差评提及率 Top 5~7 |
| 🟡 Nice-to-have 全部保留 | 未根据成本做取舍 | 设定 Nice-to-have 总预算上限（≤$2/件） |
| 🔴 单位成本超目标售价 60% | 功能过多导致成本失控 | 逐项砍减 Nice-to-have，直到毛利率 ≥30% |
| 🔴 上市时间 > 90 天 | 功能复杂导致开发周期过长 | 砍掉需要额外认证/开模的功能 |

### 4.3 优先级排序方法

当 Must-have 候选超过 7 个时，使用"影响力×可行性"打分：

| 属性维度 | 差评提及率（A） | 实现成本评分（B）| 竞品做到率（C） | 综合分 = A×40% + B×30% + C×30% |
|----------|---------------|-----------------|---------------|-------------------------------|
| [维度1] | 32%→9分 | $0.30→9分 | 20%→9分 | 9.0 ✅ 保留 |
| [维度2] | 18%→7分 | $1.50→5分 | 60%→5分 | 5.8 ✅ 保留 |
| [维度3] | 15%→6分 | $3.00→3分 | 80%→3分 | 4.2 ⚠️ 降为 Nice-to-have |

**评分标准**：
- 差评提及率：≥30%=9分, 20~29%=7分, 15~19%=6分
- 实现成本：≤$0.50=9分, $0.51~1.00=7分, $1.01~2.00=5分, >$2.00=3分
- 竞品做到率（越低=越有差异化）：<20%=9分, 20~40%=7分, 41~60%=5分, >60%=3分

## 5. 填写注意事项

### 5.1 数据来源要求

| 数据项 | 可接受来源 | 不可接受 |
|--------|----------|---------|
| 差评提及率 | `product_reviews` 分析结果 / 需求验证报告 | ❌ "大约"、"感觉" |
| 增量成本 | `ali1688_similar_product` / 1688 实际报价 / 供应商询价 | ❌ "估计加 $1 左右" |
| 竞品规格 | `product_detail` 返回数据 / 竞品 Listing 页面 | ❌ "应该是" |
| 搜索需求 | `keyword_detail` / `keyword_extends` | ❌ 无数据支撑的主观判断 |

### 5.2 常见错误

| ❌ 错误做法 | ✅ 正确做法 |
|------------|-----------|
| 规格值写"大容量" | 规格值写"20000mAh" |
| 决策依据写"用户需要" | 决策依据写"差评提及率 28%，Top1 痛点" |
| Must-have 写 10 个 | Must-have 控制在 5~7 个，其余降级或 Cut |
| 所有维度都是 Must-have | 三级分类各 ≥1 个，体现取舍 |
| 增量成本留空 | 每项标注具体金额和来源 |

## 6. 矩阵验证检查清单

- [ ] 所有品类相关属性维度已穷举？（对照竞品提取，不遗漏）
- [ ] 每个维度有且仅有一个分类（Must/Nice/Cut）？
- [ ] Must-have 数量在 5~7 个之间？
- [ ] 每个 Must-have 规格值具体可量化？（禁止模糊描述）
- [ ] 每个维度的决策依据填写完整？（含差评提及率/交叉分析结果）
- [ ] 每个维度的数据来源标注清楚？
- [ ] 增量成本已核算，总成本在毛利率允许范围？
- [ ] 竞品对比表至少包含 3 个竞品？
- [ ] Cut 列表已记录，留作 v2 迭代参考？
